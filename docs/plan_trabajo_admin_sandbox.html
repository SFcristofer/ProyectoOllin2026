0. Prerrequisitos y Respaldos

Obligatorio antes de empezar
⛔ DETENTE — Lee esto primero Varios de los cambios siguientes son irreversibles en Salesforce. Un campo borrado con datos no se recupera sin soporte de Salesforce. Sigue el checklist en orden.
Exportar TODOS los datos con DataLoader o Workbench

Exportar en CSV los siguientes objetos completos (todos los campos, sin filtros): Fianza__c, Contrato__c, Contratista__c, Beneficiario__c, Endoso__c, Vendedores__c, Tech_Relacion__c, Proyecto__c, Precalificacion__c. Guardar los CSV en una carpeta con fecha y no tocarlos.

Documentar los Page Layouts y Perfiles activos

Captura de pantalla de: Setup → Object Manager → [cada objeto] → Page Layouts. Anota qué perfiles tienen acceso a qué objetos. Esto facilita recrear los layouts en los objetos nuevos.

Desactivar todos los Flows activos del org

Setup → Flows → desactivar cualquier Flow que tenga estado "Active". Re-activarlos al final. Evita disparadores inesperados durante la migración.

Desactivar Reglas de Validación en todos los objetos involucrados

Setup → Object Manager → [objeto] → Validation Rules → desactivar. Las reglas pueden bloquear la carga masiva de datos durante la migración.

Crear un Conjunto de Cambios de respaldo (Change Set)

En el org: Setup → Deployment → Outbound Change Sets. Crear un change set que incluya todos los metadatos actuales como snapshot. No es deploy, es archivo histórico.

Avisar a todos los usuarios que el org estará en mantenimiento

Enviar comunicado a los usuarios activos. Considerar activar "System Maintenance" o simplemente coordinarlo fuera de horario laboral. Algunos módulos requieren que no haya registros creándose en paralelo.

MÓDULO 3 Reestructurar Fianza__c — cambio de relación padre ⚠️ Riesgo: CRÍTICO

Este es el cambio más importante y de mayor riesgo. Actualmente Fianza__c se vincula a contratos a través de la junction Tech_Relacion__c (campo Tipo__c = 'Contrato'). El objetivo es que Fianza__c sea hijo Master-Detail directo de Contrato__c.

⚠️ Restricción de Salesforce: no se puede cambiar Lookup a M-D si ya hay registros El proceso es: (1) crear campo M-D nuevo, (2) poblar todos los registros con el valor del padre, (3) el campo queda como M-D desde el inicio si se crea como tal y todos los registros tienen valor.
Solución: crear el campo como Lookup primero, poblar datos, después de verificar que todos tienen valor puede dejarse como Lookup (con el comportamiento de M-D logrado por permisos) o recrearse como M-D. Ver paso 4.

Proceso de creación del campo M-D Contrato__c en Fianza__c

Verificar que todos los registros de Fianza__c tienen un Contrato__c relacionado

Ejecutar en Workbench / Dev Console: SELECT Id, Name, No_Contrato__c FROM Fianza__c WHERE No_Contrato__c = null OR No_Contrato__c = ''. Si hay fianzas sin contrato, asignarles uno antes de continuar (crear un contrato "Sin Asignar" como placeholder).

Verificar que todos los Contrato__c necesarios ya existen como registros

Los números en Fianza__c.No_Contrato__c deben corresponder a registros existentes en Contrato__c. Ejecutar: SELECT Id, Name FROM Contrato__c y comparar con los valores únicos de No_Contrato__c en Fianza__c.

Crear el campo: Object Manager → Fianza__c → Fields → New → Master-Detail

Related Object: Contrato__c · Field Label: Contrato · Field Name: Contrato__c · Child RelationshipName: Fianzas · Reparenting: NO.
Salesforce advertirá que los registros existentes necesitan un valor. Aceptar y continuar.

Poblar el campo Contrato__c en todos los registros de Fianza__c con DataLoader

Preparar un CSV con columnas Id (ID de Fianza__c) y Contrato__c (ID del registro Contrato__c correspondiente). Hacer el mapeo usando el campo texto No_Contrato__c como referencia. Hacer un UPSERT sobre Fianza__c.

Verificar que TODOS los registros de Fianza__c tienen el campo Contrato__c poblado

Ejecutar: SELECT COUNT() FROM Fianza__c WHERE Contrato__c = null. El resultado debe ser 0. Si no, resolver los registros huérfanos antes de continuar.

Crear el campo Institucion__c en Fianza__c y poblar datos

Lookup a Institucion__c. Poblar usando el valor actual de Afianzadora__c como referencia para mapear al ID de Institucion__c. Usar DataLoader con el CSV de instituciones creado en el Módulo 1.

[DIFERIR] Eliminar o archivar el campo Afianzadora__c de texto

Solo después de verificar que Institucion__c está correctamente poblado en todos los registros. Antes de borrar: desactivar cualquier código Apex / Flow que haga referencia a Afianzadora__c.

Sobre Tech_Relacion__c y los Contratos Una vez que el campo M-D Contrato__c esté poblado en todas las fianzas, el vínculo Tech_Relacion__c con Tipo__c = 'Contrato' es redundante. No borrar todavía — se limpia en el Módulo 10 después de actualizar el código Apex.

Proceso de limpieza

Exportar todos los registros de Tech_Relacion__c como backup final

SELECT * FROM Tech_Relacion__c. Guardar en CSV antes de tocar nada.

Verificar en Developer Console que ninguna clase Apex activa referencia Tech_Relacion__c

Buscar en el org: Setup → Apex Classes → buscar el texto Tech_Relacion__c y Relaciones_Tecnicas. También revisar Flows activos.

Eliminar los registros con Tipo__c = 'Contrato' (ya reemplazados)

En Developer Console → Execute Anonymous: delete [SELECT Id FROM Tech_Relacion__c WHERE Tipo__c = 'Contrato'];

Evaluar si Tech_Relacion__c aún se necesita para otros tipos

Si todos los tipos están reemplazados por campos directos en los objetos, proceder a eliminar el objeto completo. Si algún tipo no tiene reemplazo, mantener el objeto y solo eliminar los campos obsoletos.

[Condicional] Evaluar el objeto Endoso__c

Una vez migrados los datos a Convenio_Modificatorio__c y Endoso_de_Fianza__c: verificar que no queden registros activos. Ejecutar: SELECT COUNT() FROM Endoso__c. Si hay registros y todos están migrados, se puede etiquetar como "Obsoleto" en el label y dejarlo en modo lectura, o eliminar después de confirmar con el equipo de negocio.

Campos a eliminar en Fianza__c (diferido — solo después de verificar data migrada)
  • Afianzadora__c (texto) → reemplazado por Lookup a Institucion__c
  • No_Contrato__c (texto libre) → reemplazado por M-D Contrato__c
Para eliminar un campo: Object Manager → Fianza__c → Fields → [campo] → Delete. Salesforce pide confirmación y lo mueve a una "papelera" de campos por 15 días antes de borrar definitivamente.

3 Vencer Recibos Pendientes Scheduled-Triggered Recibo__c Diario. Filtro: Status__c = 'Pendiente' AND Fecha_Vencimiento__c < TODAY() Cambiar Status__c → "Vencido" + Notificación in-app al vendedor
4 Calcular Comisión al Pago Record-Triggered (After Save — Update) Recibo__c Status__c cambió a "Pagado" Lookup Tabla_de_Comisiones vigente → crear Comision__c con monto calculado
5 Aplicar Convenio a Fianza Record-Triggered (After Save — Update) Convenio_Modificatorio__c Status__c cambió a "Aprobado" Actualizar Monto y/o Fecha_Vigencia en Fianza__c padre según Tipo del convenio
Tip para el Flow de Generar Recibos En el Flow de "Generar Recibos" (Flow 2), el cálculo de fecha por periodicidad requiere una Variable de tipo Date y un Loop con ADDMONTHS(). Usar una variable de iteración que incremente la fecha: Mensual = +1 mes, Trimestral = +3 meses, etc. Salesforce Flow soporta el elemento "Loop" con colecciones; crear una Colección de Recibos y al final hacer "Create Records" con toda la colección para reducir transacciones DML.
