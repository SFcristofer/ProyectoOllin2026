2. Lo que está estructuralmente MAL

Requiere corrección
⚠ Error crítico #1 — Relación Fianza → Contrato El campo No_Contrato__c en Fianza__c apunta al objeto estándar Contract (prefijo 800), no a Contrato__c custom. Además, la relación a Contrato__c custom se gestiona a través de Tech_Relacion__c junction, que es un parche. El modelo del cliente requiere un Master-Detail nativo.
⚠ Error crítico #2 — Contratista__c y Beneficiario__c como objetos custom separados El cliente define ambos como Account (estándar, Person Account para físicas). Tener objetos custom propios duplica datos, rompe la vista de 360° del cliente en Salesforce y complica integraciones futuras (Agentforce, Digital Engagement).
⚠ Error estructural #3 — Endoso__c con colisión de nombre El Endoso__c actual se usa como "Ampliación" de Fianza. El cliente define dos objetos distintos: Endoso__c hijo de Seguro__c, y Endoso_de_Fianza__c con Lookup a Convenio_Modificatorio__c. El objeto actual colisiona con ambos conceptos y bloquea el módulo de Seguros.
⚠ Error estructural #4 — Naming de Vendedor Vendedores__c usa plural, la convención estándar de Salesforce es singular: Vendedor__c. Impacta API paths, queries y labels.
⚠ Deuda técnica — Tech_Relacion__c como parche El junction object fue una solución rápida para relaciones M:N que en el modelo final son directas (M-D nativos o Lookups simples). No está en la arquitectura del cliente y genera queries complejas innecesarias. Se depreca gradualmente al implementar los M-D correctos.

7. ¿Nuevo sandbox o modificar lo existente?

Decisión estratégica

Evaluación de las dos opciones con sus ventajas, riesgos y recomendación final.

❌ Opción A — Modificar org actual
  • Cambiar No_Contrato__c (Lookup estándar → M-D custom) requiere migración destructiva
  • Mover Contratista/Beneficiario de custom a Account implica merge de registros
  • Separar Endoso__c en dos objetos rompe datos existentes
  • Añadir 7 módulos nuevos sobre base incorrecta genera deuda técnica
  • Riesgo alto de inconsistencias en datos de producción
✅ Opción B — Nuevo sandbox + migración selectiva
  • Data model correcto desde el inicio, sin compromisos
  • El sandbox actual sirve como referencia y laboratorio
  • Arquitectura Apex (Controller/Selector/Wrapper) se migra: ~20% refactor
  • 20 LWC se migran: ~30% refactor por componente (solo cambia el Apex de fondo)
  • Precalificacion__c se copia definición completa al nuevo sandbox
  • Cero riesgo para datos existentes en producción
Recomendación: Nuevo sandbox con migración selectiva de código La relación Fianza→Contrato incorrecta, los objetos custom que deberían ser Accounts, y la ausencia de 7 módulos completos hacen que construir sobre la base actual sea más costoso y riesgoso que empezar limpio con el 70% del código ya escrito reutilizable.

8. Decisión bloqueante

Confirmar con cliente antes de Fase 1
¿Contratista__c queda como objeto propio o se fusiona en Account? Esta decisión afecta la estructura de Fianza__c, Precalificacion__c y toda la UI de contratistas. Debe confirmarse antes de empezar la Fase 1.
Opción A — Contratista__c como objeto propio Contratista__c con Lookup a Account. Fianza__c → Lookup a Contratista__c. Precalificacion__c → M-D de Contratista__c. El modelo actual es cercano a esto. Permite campos propios de contratista (capacidad, antigüedad, ICP score) sin contaminar Account.
Opción B — Account puro Todo sube a Account. Precalificacion__c → M-D de Account. Más alineado con el doc del cliente pero requiere más trabajo de migración y el 360° de cliente en Account se mezcla con datos de contratista.
Segunda decisión: Proyecto__c ¿Queda como objeto propio o como campo lookup en Contrato__c? El cliente lo menciona como campo de Contrato, no como objeto. Confirmar si el alcance de los proyectos requiere objeto propio.
Tercera decisión: SPIFF Integrar, migrar o solo enlazar. Impacta el diseño de Comision__c y Tabla_de_Comisiones__c en Fase 2.
